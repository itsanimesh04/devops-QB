2-Mark Questions

2 Marks Q1. Define DevOps. ▼
DevOps is a cultural and technical movement that combines Development (Dev) and Operations (Ops) teams to shorten the software delivery lifecycle, improve deployment frequency, and ensure higher software quality through automation, collaboration, and continuous feedback.

It emphasizes breaking down silos between developers and operations teams so that software can be built, tested, and released faster and more reliably. DevOps is not merely a tool or technology — it is a philosophy and set of practices.

Key goals: Faster delivery, fewer failures, quicker recovery, and improved collaboration.

2 Marks Q2. What are the limitations of traditional software development models? ▼

Traditional models like the Waterfall model have several key limitations:

  • Siloed teams: Dev and Ops work independently, causing miscommunication and blame culture.
  • Long release cycles: Software is delivered months or years after development, making it slow to respond to market changes.
  • Manual processes: Testing, deployment, and configuration are done manually, leading to human errors.
  • Poor feedback loop: Bugs discovered late in production are expensive to fix.
  • Inflexibility: Changes in requirements are hard to accommodate mid-cycle.
  • "Works on my machine" problem: Environment inconsistencies between development and production.
2 Marks Q3. What is the CALMS framework? ▼

CALMS is a framework used to assess and guide DevOps adoption in an organization. It stands for:

  • C — Culture: Fostering collaboration, shared responsibility, and trust between Dev and Ops.
  • A — Automation: Automating repetitive tasks like builds, tests, deployments.
  • L — Lean: Eliminating waste, improving flow, focusing on customer value.
  • M — Measurement: Tracking key metrics (deployment frequency, MTTR, change failure rate).
  • S — Sharing: Open communication of knowledge, tools, and responsibilities.
2 Marks Q25. What is monitoring in DevOps? ▼
Monitoring in DevOps refers to the continuous observation of application performance, system health, and infrastructure metrics to detect issues, ensure uptime, and improve reliability.

It includes tracking logs, metrics, traces, alerts, and dashboards. Popular tools include Prometheus, Grafana, Nagios, ELK Stack, and Datadog. Monitoring closes the feedback loop in the DevOps lifecycle.

7-Mark Questions

7 Marks Q1. Explain the evolution of software development and IT operations. ▼

1. Waterfall Era (1970s–1990s)

Early software followed the Waterfall model — a sequential, phase-driven approach: Requirements → Design → Implementation → Testing → Deployment → Maintenance. Each phase had to be completed before the next began. While simple and structured, it was rigid and delivered software in 12–18 month cycles. By the time software reached users, requirements had often changed.

2. Agile Revolution (2001–onwards)

The Agile Manifesto (2001) introduced iterative development with short sprints (2–4 weeks). Teams focused on customer collaboration, working software, and responding to change. Agile fixed the development side but operations teams still deployed manually and maintained separate environments — creating a wall between Dev and Ops.

3. DevOps Movement (2008–onwards)

Patrick Debois coined the term DevOps in 2009 after the "Agile Infrastructure" talks. The idea: apply Agile and Lean principles to IT operations. Dev and Ops must collaborate. This led to continuous integration, delivery pipelines, and Infrastructure as Code.

4. Cloud-Native and Modern DevOps

With cloud platforms (AWS, Azure, GCP), containers (Docker), and orchestration (Kubernetes), DevOps evolved into cloud-native practices. Organizations now deploy hundreds of times per day (e.g., Amazon deploys every 11.7 seconds on average).

Waterfall → Agile → DevOps → Cloud-Native DevOps (Months) (Weeks) (Days) (Minutes/Hours)

Key Takeaway

The evolution has been driven by the need for speed, reliability, and collaboration. DevOps emerged as the bridge between development velocity (Agile) and operational stability, resulting in continuous delivery of value to end users.

7 Marks Q2. Explain DevOps culture, principles, and practices. ▼

DevOps Culture

DevOps is fundamentally a cultural change. It requires breaking down the "wall of confusion" between development and operations. Key cultural elements include:

  • Shared responsibility: Both Dev and Ops own the product end-to-end.
  • Blameless post-mortems: Focus on fixing systems, not blaming individuals.
  • Transparency: All teams have visibility into builds, deployments, and issues.
  • Continuous learning: Encourage experimentation and learning from failures.

Core Principles

  • Automation: Automate every repetitive task — builds, tests, deployments, infrastructure provisioning.
  • Continuous Feedback: Fast feedback loops from testing, monitoring, and users enable rapid improvement.
  • Collaboration: Dev, Ops, QA, and Security work together throughout the entire software lifecycle.
  • Iterative Improvement: Small, frequent releases reduce risk and allow fast correction.
  • Customer Focus: Delivering working software that meets user needs is the ultimate measure of success.

Core Practices

  • Version Control: All code and infrastructure changes tracked in Git.
  • CI/CD: Automated build, test, and deployment pipelines.
  • Infrastructure as Code (IaC): Infrastructure managed through code (Ansible, Terraform).
  • Monitoring and Observability: Continuous tracking of system health.
  • Microservices: Small, independently deployable services.
  • Containerization: Docker and Kubernetes for consistent environments.
7 Marks Q3. Describe the CALMS framework in DevOps. ▼

The CALMS framework was introduced by Damon Edwards and John Willis and is widely used to evaluate DevOps readiness and guide adoption.

C — Culture

Culture is the foundation of DevOps. Without cultural change, tools and processes cannot succeed. Organizations must cultivate trust, collaboration, and shared ownership. Teams should celebrate successes together and conduct blameless post-mortems when things go wrong.

A — Automation

Automation eliminates repetitive manual work. This includes CI/CD pipelines, automated testing, infrastructure provisioning (IaC), and deployment automation. Automation reduces errors, speeds up delivery, and frees engineers for higher-value work.

L — Lean

Lean principles from manufacturing (Toyota Production System) guide DevOps. The focus is on eliminating waste (waiting time, rework, manual handoffs), optimizing flow, and continuously improving processes. Work in progress (WIP) limits ensure teams don't become overwhelmed.

M — Measurement

DevOps teams track key metrics to understand and improve their performance. Key metrics (DORA metrics) include:

  • Deployment Frequency
  • Lead Time for Changes
  • Mean Time to Recovery (MTTR)
  • Change Failure Rate

S — Sharing

Sharing knowledge, tools, and experiences across teams prevents reinventing the wheel. Inner-source programs, wikis, post-mortem reports, and cross-team guilds are examples of sharing in practice.

C → Culture (People first) A → Automation (Tools & pipelines) L → Lean (Eliminate waste) M → Measurement (Data-driven decisions) S → Sharing (Collective knowledge)
7 Marks Q4. Explain the DevOps lifecycle with suitable diagram. ▼

The DevOps lifecycle is an infinite loop of continuous processes that enables rapid, reliable software delivery. It consists of 8 phases:

┌─────────────────────────────────────────┐ │ DevOps Infinity Loop │ │ │ Plan → Code → Build → Test → Release → Deploy → Operate → Monitor └──────────────────────────────────────────────┘ (Continuous Feedback)

Phase-by-Phase Explanation

  • Plan: Define features, sprints, and tasks using tools like Jira, Trello.
  • Code: Developers write code using IDEs, commit to Git/GitHub with branching strategies.
  • Build: Source code is compiled/built automatically using Maven, Gradle, npm. Jenkins triggers builds on each commit.
  • Test: Automated tests (unit, integration, regression) run via Selenium, JUnit, TestNG. Bugs are caught early.
  • Release: The release candidate is approved and tagged. This involves merge to main branch and versioning.
  • Deploy: Application is deployed to staging/production via CD pipelines. Tools: Jenkins, ArgoCD, Spinnaker.
  • Operate: Operations team ensures infrastructure is running — using Ansible, Terraform, Kubernetes to manage resources.
  • Monitor: System performance is continuously tracked using Prometheus, Grafana, ELK Stack. Alerts notify teams of issues.

The cycle then repeats — insights from monitoring feed back into planning the next sprint, completing the continuous feedback loop.

15 Marks Q1. Explain the DevOps lifecycle and DevOps toolchain in detail. ▼

Introduction

DevOps is a continuous, collaborative approach to software development and delivery. It replaces the traditional siloed model with an integrated lifecycle where development and operations teams work together throughout the entire software delivery process. The DevOps lifecycle is represented as an infinite loop (∞) to signify its continuous, iterative nature.

DevOps Lifecycle — 8 Phases

Plan → Code → Build → Test → Release → Deploy → Operate → Monitor → (back to Plan)

1. Plan

Teams collaborate to define project requirements, features, and sprint backlogs. Tools used include Jira, Confluence, Trello, Azure Boards. Agile methodologies (Scrum, Kanban) guide work breakdown. The goal is to align development with business needs.

2. Code

Developers write code on feature branches. Best practices include code reviews, pair programming, and following style guidelines. Tools: Git, GitHub, GitLab, VS Code, IntelliJ. Version control ensures traceability and rollback capability.

3. Build

Source code is compiled into executable artifacts (JAR, WAR, Docker image). Build tools like Maven, Gradle, npm, Make automate this. CI servers (Jenkins, GitHub Actions) trigger builds automatically on each commit or pull request.

4. Test

Automated testing is critical in DevOps. Types of testing include:

  • Unit Testing: Testing individual functions/methods (JUnit, pytest)
  • Integration Testing: Testing components together
  • Regression Testing: Ensuring new code doesn't break existing features
  • Performance Testing: Load and stress testing (JMeter)
  • Security Testing: SAST, DAST tools (SonarQube, OWASP ZAP)

5. Release

The tested build is versioned and tagged for release. Release management ensures approvals, changelogs, and environment-specific configurations are in place. Tools: Jenkins, GitLab CI, Nexus, Artifactory.

6. Deploy

Deployment involves pushing the application to target environments (staging → production). Deployment strategies include Blue-Green, Canary, Rolling. Tools: Kubernetes, Helm, ArgoCD, Ansible. Zero-downtime deployments are a key goal.

7. Operate

Operations teams manage infrastructure reliability, scalability, and configuration. Infrastructure as Code (Terraform, Ansible) ensures consistency. Cloud platforms (AWS, Azure, GCP) and Kubernetes handle auto-scaling and self-healing.

8. Monitor

Continuous monitoring collects metrics, logs, and traces from running applications. This phase feeds insights back to the Plan stage, completing the loop. Tools: Prometheus, Grafana, ELK Stack (Elasticsearch, Logstash, Kibana), Datadog, Nagios.

2 Marks Q4. Define Version Control System (VCS). ▼
A Version Control System (VCS) is a software tool that tracks and manages changes to source code or any set of files over time, allowing multiple contributors to collaborate and enabling rollback to previous versions.

VCS maintains a history of every change, who made it, and when. It prevents code loss, enables collaboration, and supports parallel development through branching. Examples: Git (distributed), SVN (centralized).

2 Marks Q6. What is Git? ▼
Git is a free, open-source Distributed Version Control System created by Linus Torvalds in 2005 for managing the Linux kernel source code.

Git allows developers to track file changes, collaborate on code, create branches for features, and merge changes efficiently. Every Git repository is a full copy of the entire project history. It is the most widely used VCS in the world, forming the backbone of platforms like GitHub, GitLab, and Bitbucket.

2 Marks Q7. Write the purpose of the git init command. ▼

The git init command initializes a new, empty Git repository in the current directory.

It creates a hidden .git/ subdirectory that contains all the repository metadata — object store, HEAD reference, configuration, and index. After running git init, the directory becomes a Git repository capable of tracking file changes.

Syntax: git init [directory-name]

2 Marks Q8. What is branching in Git? ▼
Branching in Git is the ability to create independent lines of development within the same repository. Each branch is a lightweight pointer to a specific commit.

Branches allow developers to work on features, bug fixes, or experiments in isolation without affecting the main codebase. Changes are later merged back. The default branch is called main (or master). Common strategy: main → develop → feature/bugfix branches.

main: A──B──────────F feature: C──D──E──┘ (merged back)

7 Marks Q5. Explain Git architecture and workflow. ▼

Git Architecture — Three Areas

Git organizes work across three logical areas:

Working Directory → Staging Area (Index) → Local Repository → Remote Repository (Untracked/Modified) (git add) (git commit) (git push)
  • Working Directory: The local file system where you edit files. Files are either untracked (new) or tracked (modified, added, deleted).
  • Staging Area (Index): A preparation zone. Files are added here with git add before committing. This allows selective commits.
  • Local Repository (.git/): The committed snapshot history. Each commit has a unique SHA-1 hash. Contains objects: blobs (file content), trees (directories), commits, tags.
  • Remote Repository: A hosted copy on GitHub/GitLab. Synced via git push and git pull.

Git Workflow

  1. Clone or init a repository: git clone <url>
  2. Create a feature branch: git checkout -b feature/login
  3. Make changes and stage them: git add .
  4. Commit the changes: git commit -m "Add login feature"
  5. Push branch to remote: git push origin feature/login
  6. Open a Pull Request (PR) for code review
  7. Merge into main after approval: git merge feature/login

Git Object Model

Git stores everything as objects: blob (file data), tree (directory structure), commit (snapshot with metadata and parent reference), and tag (named reference to a commit). Each object is content-addressed by its SHA-1 hash, making Git tamper-proof.

15 Marks Q2. Discuss the role of version control systems in DevOps and explain Git workflow. ▼

Role of VCS in DevOps

Version Control Systems are the foundation of DevOps. Every DevOps practice — CI/CD, Infrastructure as Code, Continuous Testing — depends on VCS. Without VCS, teams cannot collaborate, track changes, or automate deployments reliably.

Key roles VCS plays in DevOps:

  • Single Source of Truth: All code, configuration, and infrastructure definitions live in version control.
  • CI/CD Trigger: Every commit to a VCS triggers automated pipelines (build, test, deploy).
  • Rollback: If a deployment fails, teams can revert to the last stable commit instantly.
  • Audit Trail: Every change is documented with who, what, and when — essential for compliance.
  • Branching Strategies: Enable parallel development of features without breaking the main branch.
  • Code Review: Pull requests enforce peer review before code merges, improving quality.

Git Branching Strategies

Different teams use different branching strategies depending on their deployment frequency:

  • Git Flow: main → develop → feature/release/hotfix branches. Suitable for versioned software with scheduled releases.
  • GitHub Flow: main + short-lived feature branches. Merge via PR. Suitable for continuous deployment.
  • Trunk-Based Development: Everyone commits to main (trunk) with feature flags. Enables maximum CI.
Git Flow: main: ──A──────────────────────────E── develop: ────B──────────D────────────/ feature: ──C──────/ hotfix: ──────────/ GitHub Flow: main: ──A──────────B────────C── feature: ──X──/ ──Y──/

Complete Git Workflow in DevOps

  1. Repository Setup: git init or git clone to start. Configure user: git config --global user.name "Name"
  2. Branch from main: git checkout -b feature/user-auth
  3. Develop and stage: Edit files, then git add .
  4. Commit with message: git commit -m "feat: Add JWT authentication"
  5. Push branch: git push origin feature/user-auth
  6. Pull Request: Open PR on GitHub/GitLab. Assign reviewers.
  7. CI Pipeline triggers: Automated tests run on the PR branch.
  8. Code Review: Reviewers comment, request changes, or approve.
  9. Merge: git merge feature/user-auth into main after approval.
  10. CD Pipeline triggers: Merged code is deployed automatically to production.
  11. Tag release: git tag v2.1.0 for versioning.

2-Mark Questions

2 Marks Q9. What is Continuous Integration (CI)? ▼
Continuous Integration (CI) is a DevOps practice where developers frequently merge their code changes into a shared repository (multiple times per day), and each merge triggers an automated build and test process.

The goal is to detect integration errors quickly. If a build or test fails, the team is alerted immediately. CI ensures that the codebase is always in a testable, working state. Key tools: Jenkins, GitHub Actions, CircleCI, Travis CI.

2 Marks Q10. Define Continuous Delivery. ▼
Continuous Delivery (CD) is a practice where code changes are automatically built, tested, and prepared for release to production after passing all automated tests. The actual release to production requires a manual approval step.

It ensures the software can be deployed at any time with confidence. The difference from Continuous Deployment: in CD, a human clicks the deploy button; in Continuous Deployment, it happens fully automatically.

2 Marks Q11. What is Jenkins? ▼
Jenkins is an open-source, Java-based automation server used to implement CI/CD pipelines. It automates the building, testing, and deployment of applications.

Jenkins supports 1800+ plugins for integrating with virtually any tool in the DevOps ecosystem (Git, Docker, Maven, Kubernetes, AWS, etc.). It can be configured using a Jenkinsfile (Pipeline as Code) or through its web UI. Originally called Hudson, it was forked to Jenkins in 2011.

2 Marks Q12. What is build automation? ▼
Build automation is the process of automatically compiling source code, running tests, packaging binaries, and generating deployment artifacts without manual intervention.

It ensures consistent, repeatable builds and eliminates the "it builds on my machine" problem. Build automation tools handle dependency resolution, compilation order, and artifact creation. Examples: Maven, Gradle (Java), npm (JavaScript), Make (C/C++).

2 Marks Q13. Name any two build tools used in DevOps. ▼
  • Maven: A Java-based build tool that uses XML (pom.xml) to manage project structure, dependencies, and build lifecycle. Follows "convention over configuration."
  • Gradle: A modern, flexible build tool that uses Groovy/Kotlin DSL (build.gradle). Faster than Maven due to incremental builds and build caching. Used by Android projects.

Other notable build tools: npm (Node.js), Make, Ant, Bazel.

2 Marks Q14. What is Continuous Testing? ▼
Continuous Testing is the practice of running automated tests at every stage of the CI/CD pipeline to provide immediate feedback on business risk and software quality throughout the delivery process.

It includes unit tests, integration tests, regression tests, performance tests, and security scans — all executed automatically on every code change. Tools: JUnit, Selenium, TestNG, JMeter, SonarQube. It shifts testing "left" (earlier in the pipeline), reducing the cost of bug fixes.

7 Marks Q8. Explain the concept and benefits of Continuous Integration. ▼

Concept of CI

Continuous Integration was popularized by Martin Fowler and Kent Beck. The core idea: developers integrate code frequently (at least once per day), and each integration is verified by an automated build and test suite. The goal is to find and fix integration bugs early, when they are cheapest to correct.

How CI Works

Developer pushes code to Git ↓ CI Server detects change (webhook/polling) ↓ Fetch latest code → Compile/Build ↓ Run automated tests (Unit, Integration) ↓ Code analysis (SonarQube, linting) ↓ Build artifact (JAR, Docker image) ↓ Notify team (Pass ✓ or Fail ✗)

Key Practices of CI

  • Maintain a single source repository
  • Automate the build process
  • Make the build self-testing
  • Every commit should trigger a build
  • Fix broken builds immediately — the "stop the line" mentality
  • Keep the build fast (under 10 minutes)
  • Test in a production-like environment

Benefits of CI

  • Early bug detection: Bugs caught in minutes, not weeks.
  • Reduced integration risk: Small, frequent merges are easier to manage than large, infrequent merges.
  • Consistent build process: Same process every time, regardless of developer environment.
  • Faster feedback: Developers know within minutes if their change broke something.
  • Increased confidence: Always have a deployable version of the software.
  • Better code quality: Automated code analysis (SonarQube) enforces standards.
  • Reduced manual work: Automation replaces repetitive build and test tasks.
7 Marks Q9. Explain Jenkins architecture. ▼

Jenkins Architecture Overview

Jenkins follows a Master-Agent (Master-Slave) architecture that enables distributed builds across multiple machines.

┌────────────────────────────┐ │ JENKINS MASTER │ │ - Web UI (port 8080) │ │ - Job Scheduling │ │ - Plugin Management │ │ - Build Queue │ └──────┬──────────┬───────────┘ │ │ (SSH/JNLP) ┌──────▼──┐ ┌───▼──────┐ │ AGENT 1 │ │ AGENT 2 │ │ Linux │ │ Windows │ │ (Build) │ │ (Build) │ └─────────┘ └──────────┘

Jenkins Master

The master is the central control node. Its responsibilities include:

  • Hosting the Jenkins web UI (default: port 8080)
  • Scheduling build jobs and distributing them to agents
  • Monitoring agent health
  • Managing plugins, users, and security
  • Storing job configurations and build history

Jenkins Agent (Slave)

Agents are machines that execute build jobs delegated by the master. They can run on Linux, Windows, or macOS. Agents connect to the master via:

  • SSH: Master connects to agent using SSH credentials
  • JNLP (Java Web Start): Agent initiates connection to master

Jenkins Pipeline

Jenkins Pipelines define the CI/CD process as code in a Jenkinsfile. Two types:

  • Declarative Pipeline: Structured, easier to read and write (recommended)
  • Scripted Pipeline: Uses Groovy scripting for complex logic
pipeline { agent any stages { stage('Build') { steps { sh 'mvn package' } } stage('Test') { steps { sh 'mvn test' } } stage('Deploy'){ steps { sh './deploy.sh' } } } }

Key Components

  • Jobs/Projects: Freestyle, Pipeline, Multi-branch Pipeline
  • Plugins: 1800+ plugins for Git, Docker, Slack, AWS, etc.
  • Build Queue: Manages jobs waiting for available agents
  • Workspace: Directory where each job runs its build
7 Marks Q10. Explain the integration of Git with Jenkins. ▼

Why Integrate Git with Jenkins?

Integrating Git with Jenkins enables automated CI/CD pipelines: every code push to Git automatically triggers a Jenkins build, test, and deploy process — eliminating manual intervention and enabling rapid feedback.

Integration Methods

  • Webhook (Recommended): GitHub/GitLab sends an HTTP POST to Jenkins whenever code is pushed. Jenkins immediately triggers the job. Near-instant response.
  • Polling: Jenkins periodically checks the Git repo for changes (e.g., every 5 minutes). Less efficient than webhooks.

Step-by-Step Integration

  1. Install the Git Plugin in Jenkins (Manage Jenkins → Plugins).
  2. Create a new Jenkins Pipeline or Freestyle job.
  3. In "Source Code Management," select Git and enter the repository URL.
  4. Add Git credentials (username/password or SSH key).
  5. Specify the branch to build (e.g., */main).
  6. In "Build Triggers," select "GitHub hook trigger for GITScm polling".
  7. In GitHub repo: Settings → Webhooks → Add webhook → Jenkins URL + /github-webhook/
  8. Write a Jenkinsfile at the root of the Git repo defining the pipeline stages.

Complete Workflow

Developer pushes code to GitHub ↓ GitHub fires webhook to Jenkins (port 8080) ↓ Jenkins fetches latest code (git pull) ↓ Runs pipeline: Build → Test → Deploy ↓ Sends notification (Slack/Email) on success/failure

15 Marks Q3. Explain Continuous Integration and Continuous Delivery in detail with examples. ▼

Introduction

Continuous Integration (CI) and Continuous Delivery (CD) are two foundational practices in DevOps that together form the CI/CD pipeline — the backbone of modern software delivery. They automate the process of integrating code changes, testing them thoroughly, and preparing or deploying them to production environments.

Continuous Integration (CI) — In Detail

CI is based on the principle that developers should integrate their code changes frequently — multiple times per day — into a shared repository. Each integration triggers an automated build and test sequence.

CI Pipeline Stages:

  1. Source: Developer commits code to Git branch.
  2. Build: Jenkins/GitHub Actions compiles the code, resolves dependencies, creates artifacts.
  3. Unit Test: Automated unit tests validate individual functions (JUnit, pytest).
  4. Code Analysis: Static code analysis with SonarQube checks for code smells, vulnerabilities, coverage.
  5. Integration Test: Components tested together to verify they work as a system.
  6. Artifact Creation: Successful builds produce a deployable artifact (JAR, Docker image) stored in Nexus/Artifactory.
  7. Notification: Build results sent to Slack/email.

Continuous Delivery (CD) — In Detail

CD extends CI by automating the delivery of tested artifacts to staging and pre-production environments, making the software always deployable. A manual approval gate controls the final push to production.

CD Pipeline Stages:

  1. Deploy to Staging: Successful CI artifact is deployed to a staging environment that mirrors production.
  2. Smoke Tests: Basic tests verify the staging deployment is working.
  3. User Acceptance Testing (UAT): Product owner validates features against requirements.
  4. Performance Testing: Load tests (JMeter) verify the app handles expected traffic.
  5. Security Scanning: DAST tools (OWASP ZAP) scan the running application for vulnerabilities.
  6. Manual Approval Gate: A release manager reviews results and approves production release.
  7. Deploy to Production: One-click deployment of approved artifact.

Continuous Deployment — Fully Automated

In Continuous Deployment, the manual approval gate is removed. Every passing build is automatically deployed to production. This requires:

  • Extremely high test coverage (80–90%+)
  • Feature flags to control feature rollout independently of deployment
  • Robust monitoring and automatic rollback on failure
  • A mature DevOps culture with blameless post-mortems

What is Blue-Green Deployment?

Blue-Green Deployment is a release strategy that maintains two identical production environments — called Blue and Green — to achieve zero-downtime deployments and instant rollback capability.

┌──────────────────┐ Users ──────────▶│ Load Balancer │ └────────┬─────────┘ │ ┌──────────────┴──────────────┐ │ │ ┌─────────▼──────────┐ ┌────────────▼───────────┐ │ BLUE (v1.0) │ │ GREEN (v2.0) │ │ [LIVE - Active] │ │ [Staging - Idle] │ └─────────────────────┘ └────────────────────────┘ After switch: ┌──────────────────────────────┐ │ Switch traffic │ └──────────────────────────────┘ ┌─────────────────────┐ ┌────────────────────────┐ │ BLUE (v1.0) │ │ GREEN (v2.0) │ │ [Idle - Standby] │ │ [LIVE - Active] │ └─────────────────────┘ └────────────────────────┘

How It Works

  1. The Blue environment runs the current live version (v1.0).
  2. The Green environment is deployed with the new version (v2.0) and tested.
  3. Once Green passes all tests, the load balancer is switched to route all traffic to Green.
  4. Blue becomes idle (standby). If Green fails, switch traffic back to Blue instantly — instant rollback.
  5. When Green is stable, Blue is updated for the next release.

Advantages

  • Zero downtime: Traffic switch is instantaneous.
  • Instant rollback: Just flip the load balancer back to Blue.
  • Reduces release risk: New version tested in a production-identical environment before going live.
  • Consistent environments: Both environments are identical.

Disadvantages

  • Cost: Requires double the infrastructure (two full environments).
  • Database migrations: Schema changes must be backward-compatible during transition.
  • Not suitable for very frequent deployments (expensive standby).
7 Marks Q13. Explain Canary deployment strategy. ▼

What is Canary Deployment?

Canary deployment is a technique where a new version is released to a small subset of users first (the "canary"), while the majority continue using the stable version. Traffic is gradually shifted as confidence grows.

The name comes from the historical use of canary birds in coal mines — they were sensitive to dangerous gases and served as early warning systems. Similarly, the small "canary" user group acts as an early detector for production issues.

Stage 1: 5% canary Users ──▶ [Load Balancer] │ │ 95% ↓ 5% ↓ v1.0 servers v2.0 canary Stage 2: 30% if metrics good Stage 3: 100% — full rollout

How It Works

  1. Deploy the new version to a small number of servers (e.g., 2 out of 20).
  2. Route 5–10% of production traffic to the new version.
  3. Monitor key metrics: error rate, latency, user experience.
  4. Gradually increase traffic: 5% → 25% → 50% → 100%.
  5. If issues are detected at any stage, roll back by routing all traffic to v1.0.
  6. If all metrics are healthy, complete the rollout.

Advantages

  • Real user feedback: Tests with actual production traffic and data.
  • Lower blast radius: Issues affect only a small percentage of users.
  • Data-driven decisions: Metrics guide the rollout progression.
  • Cost-efficient: No need to maintain full duplicate environment.

Disadvantages

  • Requires sophisticated traffic splitting (e.g., via Kubernetes Ingress, Istio, or feature flags).
  • Managing two versions simultaneously adds operational complexity.
  • Database compatibility must be maintained between v1 and v2 during rollout.

Tools for Canary Deployments

Kubernetes (with Argo Rollouts), Istio, AWS CodeDeploy, Spinnaker, LaunchDarkly (feature flags).

15 Marks Q5. Explain different deployment strategies such as Blue-Green, Canary, and Rolling Deployment. ▼

Introduction

Deployment strategies define how new software versions are released to production with minimal risk and downtime. Choosing the right strategy depends on factors like application criticality, infrastructure, release frequency, and risk tolerance.

1. Recreate (Basic) Deployment

The simplest strategy: shut down v1, deploy v2. Causes downtime. Suitable only for dev/test environments or non-critical apps.

v1 [STOP] → v2 [START] (Downtime during transition)

2. Rolling Deployment

In a rolling deployment, instances of the old version are gradually replaced with the new version, one batch at a time — without taking the entire application offline.

Initial: [v1][v1][v1][v1][v1][v1] Step 1: [v2][v2][v1][v1][v1][v1] (2 updated) Step 2: [v2][v2][v2][v2][v1][v1] (4 updated) Step 3: [v2][v2][v2][v2][v2][v2] (all updated)

Advantages: No extra infrastructure needed; gradual transition; easy with Kubernetes (RollingUpdate strategy).
Disadvantages: Both v1 and v2 run simultaneously (session handling issues); slower rollout; rollback is slow (must re-roll all instances).

Kubernetes Rolling Update: spec: strategy: type: RollingUpdate rollingUpdate: maxUnavailable: 1 # max pods down at a time maxSurge: 1 # max extra pods during update

3. Blue-Green Deployment

Maintains two identical environments (Blue=current, Green=new). Traffic is switched atomically via load balancer.

Blue (v1) ──[Active]──▶ Users Green (v2) ──[Staging, tested] After switch: Blue (v1) ──[Idle, rollback ready] Green (v2) ──[Active]──▶ Users

Advantages: Zero downtime; instant rollback; production-identical test environment.
Disadvantages: Double infrastructure cost; database migration complexity.

4. Canary Deployment

Gradually shifts traffic from old to new version, monitoring metrics at each step before proceeding.

Phase 1: v1=95%, v2=5% → Monitor errors/latency Phase 2: v1=75%, v2=25% → Metrics healthy? Continue Phase 3: v1=50%, v2=50% → A/B comparison Phase 4: v1=0%, v2=100% → Full rollout complete

Advantages: Real user testing; small blast radius; data-driven rollout decisions.
Disadvantages: Requires traffic splitting infrastructure; version compatibility during rollout.

5. Feature Flag (Dark Launch) Deployment

Code is deployed but new features are toggled on/off via feature flags, independently of deployment. Allows deploying code without exposing features to users.

if (featureFlag.isEnabled("new_checkout", user)) { showNewCheckout(); } else { showOldCheckout(); }

Choosing the Right Strategy

  • High-traffic, critical app with strict SLA: Blue-Green
  • Need to test with real users before full rollout: Canary
  • Limited infrastructure budget, stateless app: Rolling
  • Need to decouple deployment from release: Feature Flags

Conclusion

Modern DevOps organizations often combine strategies — for example, using Canary + Feature Flags together. Kubernetes natively supports Rolling and Canary (via Argo Rollouts), while cloud providers (AWS, Azure, GCP) offer managed Blue-Green deployment services. The key goal of all strategies is to minimize user impact while maximizing deployment frequency and confidence.

2 Marks Q15. Define Infrastructure as Code (IaC). ▼
Infrastructure as Code (IaC) is the practice of managing and provisioning infrastructure (servers, networks, databases, load balancers) through machine-readable configuration files instead of manual processes or interactive configuration tools.

IaC treats infrastructure the same way developers treat application code — version-controlled, testable, and repeatable. 2 Marks Q16. What is configuration management? ▼
Configuration Management (CM) is the process of systematically managing, tracking, and controlling the configuration of software, servers, and infrastructure components to maintain consistency and desired state over time.

CM tools ensure that servers are configured identically (no "snowflake servers"), automatically apply configurations at scale, and enforce compliance. They use a desired state model — you declare what the system should look like, and the tool makes it so. Tools: Ansible, Chef, Puppet, SaltStack.

2 Marks Q18. What is Terraform? ▼
Terraform is an open-source IaC tool developed by HashiCorp that allows you to define and provision cloud infrastructure using a declarative configuration language called HCL (HashiCorp Configuration Language).

Terraform is cloud-agnostic — it supports AWS, Azure, GCP, and hundreds of other providers through a plugin system. It maintains a state file to track deployed resources and can plan changes before applying them. Key commands: terraform init, terraform plan, terraform apply, terraform destroy.

7 Marks Q14. Explain Infrastructure as Code with example. ▼

Why IaC?

Traditionally, infrastructure was provisioned manually — logging into a console or running scripts without version control. This led to:

  • Configuration drift: Servers gradually differ from each other due to manual changes.
  • Snowflake servers: Servers so customized they can't be reproduced.
  • No audit trail: No record of what changed, when, and why.
  • Slow provisioning: Setting up a new environment takes days or weeks.

IaC solves all these problems by treating infrastructure configuration as source code.

IaC Approaches

  • Declarative (What): You describe the desired end state; the tool figures out how to achieve it. Example: Terraform, CloudFormation. Idempotent by nature.
  • Imperative (How): You write step-by-step instructions. Example: Bash scripts, Ansible tasks in procedural mode.

2 Marks Q19. Define containerization. ▼
Containerization is an OS-level virtualization method that packages an application and all its dependencies (libraries, runtime, config files) into a single, portable unit called a container.

Containers share the host OS kernel but run in isolated namespaces, making them lightweight (MB vs. GB for VMs) and fast to start (seconds vs. minutes). They guarantee the application runs the same way regardless of the environment — development, testing, or production. The leading containerization platform is Docker.

2 Marks Q20. What is Docker? ▼
Docker is an open-source containerization platform that enables developers to build, package, ship, and run applications as lightweight, portable containers.

Released in 2013 by Docker Inc., it uses Linux kernel features — namespaces (isolation) and cgroups (resource limits) — to create containers. Docker solved the classic "works on my machine" problem by bundling the app with its environment. Key components: Docker Engine, Docker Hub, Docker Compose, Docker Swarm.

Docker Architecture (Detailed)

Docker's architecture is built on the Linux kernel's namespaces and control groups (cgroups):

  • Namespaces: Provide isolation — pid (process), net (networking), mnt (file system), uts (hostname), ipc, user namespaces.
  • cgroups: Control resource allocation (CPU, memory, I/O) per container.
  • Union File System (OverlayFS): Docker images use layered file system. Each Dockerfile instruction creates a new read-only layer. The container adds a writable layer on top.

Kubernetes (K8s) is an open-source container orchestration platform originally developed by Google (released 2014) that automates the deployment, scaling, load balancing, self-healing, and management of containerized applications.

K8s manages clusters of nodes, scheduling containers (in Pods) onto available resources. 2 Marks Q22. Define Pod in Kubernetes. ▼
A Pod is the smallest deployable unit in Kubernetes. It represents one or more tightly coupled containers that share the same network namespace (IP address and ports), storage volumes, and lifecycle.

Containers within a Pod communicate via localhost. Pods are ephemeral — they are created, destroyed, and replaced by Kubernetes automatically. In most cases, a Pod contains a single container. Multi-container Pods are used for sidecar patterns (e.g., logging sidecar, proxy sidecar with Envoy/Istio).

Control Plane Components

  • kube-apiserver: The front-end REST API for the cluster. All commands (kubectl) go through it. It validates and processes API requests.
  • etcd: Distributed key-value store — the single source of truth for all cluster state data (what pods exist, their status, etc.).
  • kube-scheduler: Watches for newly created Pods with no assigned node. Selects the best worker node based on resource requirements, affinity rules, and constraints.
  • kube-controller-manager: Runs controller loops that ensure the actual state matches the desired state. Includes Node Controller, ReplicaSet Controller, Deployment Controller, etc.
  • cloud-controller-manager: Integrates with cloud provider APIs (AWS, GCP, Azure) for load balancers, storage, and node management.

Worker Node Components

  • kubelet: Agent running on each node. Receives Pod specs from the API server and ensures containers are running correctly. Reports node/pod status.
  • kube-proxy: Network proxy that maintains network rules. Enables communication to Pods from inside or outside the cluster using iptables/IPVS rules.

1. Pods

Pods are the basic unit of deployment. A Pod spec defines container image, ports, resource limits, environment variables, and volume mounts.

2. Services

Pods are ephemeral with dynamic IPs. A Service provides a stable network endpoint (DNS name and IP) that routes traffic to matching Pods via label selectors. Types:

  • ClusterIP (default): Internal cluster access only. Used for inter-service communication.

  • LoadBalancer: Provisions a cloud load balancer (AWS ELB, GCP LB). External access via LB IP.
  • ExternalName: Maps service to a DNS name (e.g., an external database).

2 Marks Q24. What is DevSecOps? ▼
DevSecOps (Development + Security + Operations) is the practice of integrating security at every phase of the DevOps lifecycle — rather than treating it as a separate, end-of-pipeline activity.

It is based on the principle of "Shift Left" security — moving security checks earlier in the development process (to the left on the timeline), where fixing issues is cheaper and faster. DevSecOps makes security a shared responsibility of Dev, Sec, and Ops teams. Tools include SAST (SonarQube), DAST (OWASP ZAP), container scanning (Trivy), and secrets detection (GitLeaks).

Traditional Security Problem

In traditional development, security was a "gate" at the end — security teams reviewed code just before release. This meant:

  • Security issues discovered late are expensive to fix
  • Security became a bottleneck slowing releases
  • Developers saw security as "not their problem"
  • Attackers exploited the long window of vulnerability

The DevSecOps Approach — Shift Left

DevSecOps embeds security into every phase of the CI/CD pipeline:

Plan → Threat modeling, security requirements review Code → SAST (Static Analysis), pre-commit hooks, IDE security plugins Build → Dependency scanning (OWASP Dependency-Check, Snyk) Test → DAST (Dynamic Analysis, OWASP ZAP), Penetration testing Release → Image scanning (Trivy, Clair), Signing (cosign) Deploy → Runtime security (Falco), network policies, RBAC Operate → WAF, IDS/IPS, SIEM (Splunk), secrets management (Vault) Monitor → Security event monitoring, anomaly detection, compliance

Key DevSecOps Practices

  • SAST (Static Application Security Testing): Analyzes source code without executing it to find vulnerabilities. Tools: SonarQube, Checkmarx, Veracode. Integrated into CI pipeline.
  • DAST (Dynamic Application Security Testing): Tests the running application for vulnerabilities (SQL injection, XSS, CSRF). Tools: OWASP ZAP, Burp Suite.
  • SCA (Software Composition Analysis): Scans third-party libraries and dependencies for known CVEs. Tools: Snyk, OWASP Dependency-Check, WhiteSource.
  • Container Security: Scan Docker images for OS and package vulnerabilities. Tools: Trivy, Clair, Anchore.
  • Secrets Management: Prevent hardcoded credentials. Tools: HashiCorp Vault, AWS Secrets Manager, GitLeaks (scan repos for secrets).
  • Infrastructure Security: Scan IaC files (Terraform) for misconfigurations. Tools: Checkov, TFSec, Terrascan.
  • Compliance as Code: Automated compliance checks (CIS benchmarks, PCI-DSS, HIPAA) using tools like InSpec, OpenSCAP.

Security Culture in DevSecOps

DevSecOps is not just about tools — it requires cultural change: developers must be trained in secure coding practices (OWASP Top 10), security must be included in sprint planning, and security metrics must be tracked alongside delivery metrics.

  • Pod Security Standards: Enforce non-root containers, read-only file systems
  • Network Policies: Restrict Pod-to-Pod communication
  • Secrets Management: Use HashiCorp Vault or K8s Secrets (base64-encoded, not plaintext)
  • IaC Scanning: Scan Kubernetes manifests with Checkov or Kubesec

Stage 7: Operate & Monitor (Runtime Security)

  • Falco: Cloud-native runtime security — detects anomalous container behavior (e.g., unexpected shell spawned in container)
  • WAF (Web Application Firewall): AWS WAF, Cloudflare — filters malicious web traffic
  • SIEM: Splunk, IBM QRadar — aggregate and correlate security events
  • Intrusion Detection: Snort, Suricata — detect network-level attacks
  • Compliance Monitoring: AWS Security Hub, Azure Defender — continuous compliance checks
